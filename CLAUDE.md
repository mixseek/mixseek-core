
# プロジェクトガイドライン

## プロジェクト概要

mixseekは、LLMを活用したマルチエージェントオーケストレーションフレームワーク。
リーダーエージェントと複数のメンバーエージェント構成を1チームとし、複数チームが指定された単一のタスクにコンペティション形式で取り組む。
各チームはラウンドごとにタスクへの回答を提出し、提出物はEvaluatorによる評価を受けてスコア化され、リーダーボードに掲載される。
各チームは評価フィードバックを受け、次のラウンドで回答を洗練させることを繰り返す。

## 憲章

### I. 日本語優先

すべてのアウトプット（ユーザとの会話、仕様書・計画書などのMarkdown成果物、実装コードのdocstringやコメント）は**必ず日本語**で記述しなければならない。これにより、プロジェクト全体の一貫性と日本語話者にとっての可読性を確保する。

**ただし、コードの識別子（変数名、関数名、クラス名、テストメソッド名等）は日本語化しない。** PEP 8に従い英語の`snake_case`/`PascalCase`で記述すること。日本語での説明はdocstringやコメントに記載する。

### II. 可読性最重視（NON-NEGOTIABLE）

コードとドキュメントの**可読性を最優先**する。短縮形や難解な実装より、明確で理解しやすいコードを選択すること。

**必須事項**:
- 変数名・関数名は意図を明確に表現する（略語の使用は最小限に）
- 複雑なロジックには日本語コメントを必須とする
- コードレビューにおいて可読性が最重要な評価基準となる
- 「動けばよい」ではなく「読んで理解できる」ことを優先

### III. テスト駆動開発（TDD）（NON-NEGOTIABLE）

テスト駆動開発を**厳守**する。Red-Green-Refactorサイクルを必ず守ること。

**必須事項**:
- 実装前に必ずテストを作成する
- テストが失敗することを確認してから実装を開始する
- 包括的なテストカバレッジを維持する（カバレッジ目標: 80%以上）
- モックやスタブを適切に利用し、テスタビリティを確保する

### IV. 品質ゲートの遵守（NON-NEGOTIABLE）

実装時は**makeによる品質チェックを常に実行**し、コード品質を維持しなければならない。

**必須事項**:
- commit前に必ず以下のコマンドを実行し、必要な修正を行う
    - `make -C dockerfiles/ci quality-gate-fast`: CI 相当のフル実行（ruff lint + format check + mypy + tests）。
- 必要に応じて、`uv run` により個別にruff、mypyチェックを行う

### V. パッケージ構造の明確化

Pythonプロジェクトでは**トップレベルパッケージを使用**し、名前空間の衝突を避け、import文を明確にする。

**必須事項**:
- `src/` 配下にプロジェクト固有のトップレベルパッケージを配置する
- すべてのディレクトリに`__init__.py`を配置し、パッケージとして明示する
- 一般的な名前（`clients`, `config`, `utils`等）は必ずトップレベルパッケージ配下に配置する
- import文は絶対importを使用する（例: `from package.clients.base import BaseClient`）
- パッケージとして配布可能な構造を維持する（`pyproject.toml`での管理）

### VI. DRY原則（Don't Repeat Yourself）（NON-NEGOTIABLE）

コードの重複を排除し、**保守性と美しさを最大化**する。同じロジックや知識は単一の場所に定義すること。

**必須事項**:
- 同じコードを複数箇所にコピーペーストしない
- 共通ロジックは関数・クラス・モジュールとして抽出する
- 設定値や定数は一箇所で管理する
- ビジネスルールの重複を避ける（単一責任の原則）
- 抽象化が過剰にならないよう注意する

### VII. ファイル・関数サイズ制限

ファイルや関数が長くなりすぎないようにし、認知負荷を下げることを重視する。
ファイルや関数が長いと感じたら、関心事の境界で分割を検討すること。
ただし、行数を減らすことを目的とした機械的な分割は避ける。

**必須事項**
- 1つのファイルや関数が長くなりすぎて見通しが悪いと感じたら、分割を検討する
- 分割する際は行数ではなく関心事の境界を基準にする


## コミットメッセージ規約

本プロジェクトでは、自動バージョン管理およびCHANGELOG生成のため、すべてのコミットに [Conventional Commits](https://www.conventionalcommits.org/) 形式を**必須**とする。

### フォーマット

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 必須タイプ一覧

```
feat:     新機能の追加（安定版ではマイナーバージョン、プレリリースではビルドバージョンが上がる）
fix:      バグ修正（パッチバージョンが上がる）
docs:     ドキュメントのみの変更
refactor: リファクタリング（機能変更なし）
test:     テストの追加・修正
release:  バージョン更新・リリース（commitizenが使用）
```


## ツール使用時の注意

### 一括置換の禁止

sedコマンドなどによる実装ファイルの一括置換は想定外のバグを生むため、絶対に使用しない。
